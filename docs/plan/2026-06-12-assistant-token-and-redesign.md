## 一、token 账本(实测,非估算)

数据源:示例项目事件日志 `agent.turn.finished.usage`(2026-06-12 真机走查两轮)。

| 轮 | 请求数 | promptTokens | completionTokens | 合计 |
|---|---|---|---|---|
| 轮1 拆镜头(create+connect) | ~3 | 28,250 | 890 | 29,140 |
| 轮2 改提示词(被锁 deny) | 2 | 22,685 | 585 | 23,270 |

**97.5% 是输入。** 每次 LLM 请求 ≈11.3k 输入,构成(按代码与磁盘快照实测拆解):

| 块 | 大小/请求 | 来源 | 性质 |
|---|---|---|---|
| ① 工具 schema(6 个 zod→JSON) | ~2.5k | AI SDK 每请求重发 | 固定 |
| ② 当轮用户消息 = 指令(0.7k)+**画布快照全字段 pretty JSON**(2-3k,样例项目实测 1.9k,随节点线性涨;真实项目带 result/history/runs 会更大)+**选中节点整份重复**+模型清单(0.4k)+记忆(0.2k) | ~4-4.5k | `buildGenerationCanvasAgentPrompt` | 每轮重建 |
| ③ 历史:上轮 displayPrompt(短,✓ 旧快照不累积——agentChatV2:500 已做对)+上轮**全部 tool args/results 原文**(3 节点长提示词全文回放) | ~3.5-4k | session history | 随轮数涨 |

**两个结构性放大器**:
1. **步数乘法**:一轮 N 次工具往返 = N 次全上下文重发(轮1 三发 → 28k)。
2. **缓存击穿**:动态快照拼在用户消息**开头**、每轮重建;`system` 槽空着 → vendor 自动前缀缓存(OpenAI 系对稳定前缀自动半价/免重算)永不命中。

### 修复路线(独立切片,不混进 UI 重设计)

| 刀 | 做法 | 预期 |
|---|---|---|
| T1 快照瘦身 | 投影白名单(id/kind/title/prompt 截 60/status/locked + 边紧凑行),不 pretty-print;选中节点只传 id 列表 | 每请求 -2~3k |
| T2 稳定前缀进 system | 身份/规则/工具说明/模型清单挪 `system`(跨轮 byte 稳定),快照挪用户消息**尾部** | OpenAI 系前缀缓存命中,输入计费~减半 |
| T3 旧轮工具回放截断 | capAgentHistory 对 ≥2 轮前的 tool args 只留 head | 长会话不再线性涨 |

预期:同样旅程 29k → **8-10k**,缓存命中后计费再降一半。

## 二、用户可见信息盘点(R2:每条问「有行动价值吗」)

生成区助手面板现状 **21 类可见信息、4 套卡片语言**:

| # | 信息 | 现状位置 | R2 裁决 |
|---|---|---|---|
| 1 | 「助手」标题+logo | header | 留 |
| 2 | 新对话/收起 | header | 留 |
| 3 | **「6 个工具 ⌄」fold** | header 下常驻一行 | **✂ 收纳**——用户不能改工具集,0 行动价值,白占一行 |
| 4 | **「AI 记得 N 条 ⌄」fold** | 再一行 | **收纳**——高价值但低频,不该常驻两行 |
| 5 | 开场引导卡+3 建议 chips | 空会话 | 留(冷启动价值高) |
| 6 | 用户/AI 气泡 | thread | 留 |
| 7 | 「以上对话 AI 已不再记得」分隔线 | thread | 留(诚实义务) |
| 8 | **「已执行 N 个工具调用。」** | 拼在 AI 正文里 | **✂ 删**——回执卡已表达,正文拼接是双重陈述 |
| 9 | **「本轮 ~29.1k tokens」** | 每条 AI 气泡脚注 | **降级进审计层**——本次提问就是证据:裸数字无行动价值,只制造焦虑;S7 有金额后再上浮 |
| 10 | 「只说不做」换模型提示 | AI 正文 | 留(高行动价值)但样式统一 |
| 11 | 计划卡(节点行+模型 chips+可编辑 prompt+确认全部) | thread | 留——这是面板的主角 |
| 12 | **单工具 pending 卡:`connect_canvas_edges` 原文+raw JSON** | thread | **✂ 重做**——内部词汇+JSON 直怼用户脸上,与计划卡完全两套语言 |
| 13 | committed 卡「✓ 已应用…查看步骤/整笔撤销」 | thread | 留,但与 #14 合并成一种「回执」 |
| 14 | 偏差卡「⚠ 执行与批准有 N 处出入」 | thread | 并入回执(出入=回执的异常态,不是另一种卡) |
| 15 | 选择性撤销确认(列明将丢失) | committed 卡内 | 留 |
| 16-19 | composer:附件/textarea/模式▾/模型▾/发送 | 底部 | 留(已合理) |
| 20 | toast「AI 已应用·整笔撤销」 | 画布侧 | 留(第二入口是拍板项) |
| 21 | BatchPlanOverlay 波次确认条 | 画布原位 | 留(S2b 拍板方案 A) |

**病根一句话:对话流里混了 4 套卡片语言(计划卡/raw-JSON 工具卡/committed 卡/偏差卡)+ 2 条常驻 fold + 2 种正文拼接的状态陈述——没有一条统一的「信息在哪一层」规则。**

## 三、整体设计:渐进披露三层(方案 A,推荐)

**一条规则:对话流里只允许三种东西——你说的 / AI 说的 / 一张等你点头的「提议卡」(及其确认后的一行「回执」)。其余一切收进两个固定入口。**

```
┌─ header ──────────────────────────────┐
│ 助手        [◉ 项目脑]   [新对话] [×] │ ← 记忆/能力/会话健康 唯一入口(有新记忆时蓝点)
├─ thread ──────────────────────────────┤
│  用户气泡                              │
│  AI 气泡(纯文字,无统计拼接)            │
│  ┌ 提议卡(唯一卡式) ────────────┐      │ ← 计划/改词/删除/批量生成 全用同一张
│  │ ▍创建 3 个镜头节点            │      │   人话标题,动词开头
│  │  1 清晨京都小巷 [Seedream·16:9]│      │   内容行按提议类型变化
│  │  2 骑车穿过鸟居 [Seedream·16:9]│      │   prompt 行内可编辑(现有交互保留)
│  │  …                  拒绝│确认  │      │
│  └──────────────────────────────┘      │
│  ✓ 已应用 · 创建 3 节点  撤销 ▸详情    │ ← 回执行(确认后卡片塌缩成一行)
│                                        │   ▸详情 = 步骤+对账+本轮用量(审计层)
│  ⚠ 有 2 处与批准不符  详情│撤销        │ ← 出入 = 回执的异常态(同一行变色),非第五种卡
├─ composer ────────────────────────────┤
│ [📎] 告诉我画布上想怎么搭…             │
│ 模式 Agent▾  GPT-5.5▾            [↑] │
└────────────────────────────────────────┘
```

**层定义**:
- **L1 对话流**(默认可见):气泡 + 提议卡 + 回执行。零统计、零内部词汇、零 JSON。
- **L2 项目脑**(header 入口,点开浮层):记忆 N 条(pin/改/删)/能力说明(原「6 个工具」)/「AI 已不再记得早先 N 轮」状态。有新记忆时入口亮蓝点——记忆从「常驻两行」变「有事才示意」。
- **L3 审计层**(回执行「▸详情」展开):步骤列表 + 对账结果 + 本轮 token/将来金额 + 事件下钻。tokens 数字归这里。

**统一提议卡的内容行词表**(杀 toolName 原文):
| 工具 | 标题 | 内容行 |
|---|---|---|
| create_canvas_nodes | 创建 N 个节点 | 节点行(现计划卡样式) |
| connect_canvas_edges | 连接 N 条引用线 | 「A → B」行(人话节点名,非 clientId) |
| set_node_prompt | 改写「X」的提示词 | 新旧词对照(diff 样式,复用偏差卡行) |
| delete_canvas_nodes | 删除 N 个节点 | 节点名行,红字 |
| run_generation_batch | 批量生成 N 个节点(将产生费用) | 波次行「第1波:A,B → 第2波:C」 |
