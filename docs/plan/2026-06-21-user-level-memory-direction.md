# 用户级 / 跨项目记忆 — 方向探讨（只聊清楚，未开工）

> 2026-06-21。用户问「我们要做记忆库吗？记了用户哪些东西？分层记忆想过吗？」并要求「搜资料看时间点、别吃老本」。本文档是**思考产物 + 将来拍板的依据**，不是实现单。结论先行：**Nomi 目前只有项目级记忆，缺的是「用户级」那一层；该不该做、做成什么样，这里把边界和取舍摊开。**

---

## 0. 一句话立场

要做用户级记忆，**别抄大厂的「后台自动综合 dossier」**（它后台推断并悄悄改写你的画像，跟我们的精度铁律 + 诚实交付冲突，2026 一手也点了新风险）。要做就做**「显式 + 可看可编辑」的薄用户层 + 跨项目可复用资产（基础库）**，effect-first、不 auto 魔法、本地优先当招牌。**但这是可延后的一层**——项目级脊梁刚立，先用扎实。

---

## 1. 现状：Nomi 记了用户哪些东西（诚实盘点）

**目前只有「项目级」，没有「用户级」。**

| 层 | 记什么 | Nomi 现状 |
|---|---|---|
| **会话级**（working） | 当前对话上下文 | ✅ 工作缓存（切片0，per-area sessionKey 落盘） |
| **项目级**（project） | 设定卡（角色/场景）/ 锁 / 项目偏好（overrides）/ 软偏好（user-added） | ✅ 刚立的脊梁（projectMemory + 设定卡渲染 + 挂载徽章 + 软偏好写入路径） |
| **用户级**（user-profile） | 跨项目：你惯用的风格、常用模型、品牌（NomiWordmark 这类）、命名习惯 | ❌ **没有** |
| **基础库**（可复用资产） | 跨项目复用的设定卡 / 风格卡 / 提示词 | ⚠️ 提示词库已有（并行会话交付）；设定卡/风格跨项目复用 ❌ 没有 |

**刻意的设计**：projectMemory 一直 per-project 隔离（大量防串台守卫）。早前用户澄清过「多会话历史库=伪需求，真实需求是打开旧项目接着干」，所以跨项目记忆一直延后。

## 2. 近期调研（2025 H2–2026，带日期，别吃老本）

> 完整来源见调研记录。关键是「现在 vs 一年前」的变化——我之前的认知有过时风险。

**当前主流做法（2026 中）**：一个**异步后台进程自动横跨全部历史综合出一份"活的"用户记忆**，**自我覆盖、有时间感知**（过期事实自动改写）、**不靠用户手动管**，配**可看/可编辑 summary + 来源溯源**；记忆**按项目/workspace 默认隔离**当安全护栏；存储用 **entity-centric 用户画像层 + episodic/semantic + 可选时序知识图**多层。
- OpenAI **Dreaming V3（2026-06）**、Claude **项目级 24h 综合（2025-09起）**、Gemini **Personal Context（2025-08）**、Mem0/Zep/Letta（2026 Q1-Q2）。

**相比一年前变了什么（过去 6–12 个月的实质新机制）**：
1. **手动便签 → 后台自动综合**（2024 末三家还是「叫它记才记」）。
2. **新增时间维度**：记忆自更新、过期事实自动改写（Dreaming V3 招牌 / Zep validity 窗）。
3. **跨厂可搬迁**（2026 出现 memory 导入/导出）。
4. **学术分层换框**：2026-02 综述从「认知五层」→「结构优先」四类，**把用户画像单列一层**；并出现反认知类比实证（Letta 纯 filesystem 反超向量库 74%——「别迷信认知类比，平铺文件可能就够」）。

**对 Nomi 最 load-bearing 的两条**：
- ✅ **Claude 刻意做项目隔离当安全护栏** → **验证了我们的 per-project 隔离是对的**，不是落后。
- ✅ 创作工具（Adobe Firefly Projects/Elements/Style Kit，2025-2026）**记的是「偏好 + 可复用资产」，刻意不把内容喂进基模**；个性化=学偏好+复用资产，≠ 训练你的内容。隐私靠**可看可编辑 summary + 内容溯源**。本地优先在 2026 是隐私**招牌**。

**2026 点名的新风险（auto-synthesis ≠ 纯收益）**：静默失败（memory blindness、写入格式错被吞）、context bleed（账号内跨场景串味）、prompt-injection 经记忆做外泄通道、**自更新记忆可审计性下降**（要审的不只是你说过什么，还有它推断并悄悄改写了什么）。→ 这些都撞我们的「界面不许说谎 / 精度铁律」。

## 3. 如果做用户级，做成什么样（推荐形态）

**不是** 大厂那种后台自动推断的 dossier。**是** 一个**显式、可看、可编辑、本地**的薄用户层，分两块：

### 3.1 用户画像层（user-profile）—— 稳定偏好
- 记**跨项目稳定的偏好**：惯用画风（如「冷色蓝灰」）、常用模型（如 Seedance/某 text 大脑）、品牌（NomiWordmark/配色）、节奏/时长习惯、命名习惯。
- **写入 = 显式或提议态**：用户在软偏好提议卡上确认「记住」时，可选「只这个项目 / 所有项目」；或在设置里手动维护。**禁止后台自动推断改写**（守精度铁律）。
- **可看可编辑**：一个「关于你」的小面板，逐条可改可删，origin 标清（你加的 / AI 提议你确认的）。
- **注入**：跨项目时拼进 system（和项目记忆同一注入点，切片0 的共享 block 天然能扩）。

### 3.2 基础库（reusable assets）—— 跨项目复用的设定/资产
- 设定卡（角色/场景/风格卡）能**收藏进基础库 → 在新项目引用**（拷贝语义，不是全局共享真相源——避免一处改处处变的串台）。
- 风格模板 / 提示词模板（提示词库已有雏形，可并入）。
- **版权红线**：纯文本 prompt/风格模板安全可做；参考图素材库需用户提供合规源（沿用既有判断）。

## 4. 边界：记什么 / 不记什么（2026 共识 + 我们的铁律）

| 记 | 不记 |
|---|---|
| 稳定偏好（风格/模型/品牌/节奏） | 逐条原始对话内容当画像 |
| 可复用资产（设定卡/风格/提示词模板） | 把你的内容喂进任何训练 |
| 用户显式确认 / 提议态转正的事实 | AI 后台自动推断并悄悄改写的「画像」 |
| origin 标清、可看可编辑、可删 | 不可审计的黑盒 dossier |
| 默认 per-project；跨项目要显式「记到所有项目」 | 默认全局自动串味（context bleed） |

身份/外观一致性仍走**生成参数（参考图/seed/权重）**，不进文本记忆（贯穿铁律，前文设定卡结论不变）。

## 5. 取舍对比（拍板用）

| 方向 | 用户看到什么 | 代价 | 适配 Nomi |
|---|---|---|---|
| **A 先不做用户级** | 维持现状（项目级脊梁）| 0；跨项目不「越用越懂你」| 高（D2 solo·广度是敌人；项目隔离被 2026 验证对）|
| **B 轻量用户层（推荐若做）** | 「关于你」可编辑偏好面板 + 设定卡能跨项目复用 | 中：新建 user-profile 存储（非 per-project）+ 提议卡加「记到所有项目」+ 基础库引用 UI | 中-高（effect-first + 显式可审计 + 本地招牌，最像用户会选的解）|
| **C 完整 auto-synthesis 用户记忆** | 「它自动越来越懂我」| 大：后台综合引擎 + 自更新 + 审计兜底；撞精度铁律 + solo 过度 | 低（不推荐）|
