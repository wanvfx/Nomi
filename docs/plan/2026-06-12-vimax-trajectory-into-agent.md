## 一、ViMax 在做什么（技术讲解）

长视频 = 多镜头叙事，难点是跨镜头一致性（角色长相/场景几何漂移）。ViMax 的解法四件套：

1. **图依赖生成（§2.3.1，本方案核心）**：规划期分析所有镜头描述，找出共享视觉元素（角色/场景/道具）→ 构建**依赖图** → 生成期按**拓扑序**执行：无依赖的镜头并行，有依赖的镜头把前置节点的产物（图/视频）作为**参考条件**喂进生成（`V_i = GenConditioned(T_i, {V_j : j ∈ Prerequisites(i)})`）。
2. **两步走文生视频（§2.1.1）**：先生成关键图（角色定妆照、场景底图、镜头关键帧——关键帧生成时引用角色/场景图做 image-to-image），再用关键帧 image-to-video 出每镜头视频。
3. **过渡视频锚定空间（§2.3.2）**：同场景多机位时，生成一段「从机位 A 摇到机位 B」的过渡视频，从中抽帧作为各机位的空间一致性锚——这是「视频作为下一个视频的参考」的具体机制。
4. **VLM best-of-k 质检（§2.1.2）**：每个任务并行采样 k 个候选，VLM 按镜头规格打分，最高分进入下一环节。

另有：层级故事分解 + RAG 全局上下文（§2.2）、多 agent 分工（编剧/分镜/定妆/生成/质检）。

## 二、和 Nomi 的关系（通俗讲解）

ViMax 的依赖图是**藏在管线里的内部结构**；Nomi 的生成画布**本来就是一张用户看得见、改得动的依赖图**（节点 + 引用边 + 已有的「依赖波次」调度器 `buildDependencyWaves`——参考先生成已经实现）。所以我们不需要抄它的管线，而是把它的**轨迹思维**装进现有画布 Agent：

> 用户说「把这个故事做成视频」→ Agent 规划的不再是「7 个镜头图节点」这一层，而是一张完整轨迹：
> **角色/场景参考图 → 各镜头关键帧（引用参考图）→ 各镜头视频（引用关键帧 + 前一镜头）**，
> 一次批准，按波次执行，参考没好的镜头自动等待。

## 三、差距盘点（现状 → 要补什么）

| # | ViMax 能力 | Nomi 现状 | 差距 |
|---|---|---|---|
| G1 | 计划即轨迹（三层 DAG：参考→关键帧→视频） | agent 拆镜头只建单层 image 节点 + 顺序边（c493929 后节点+边已原子化） | **prompt/工具升级**：拆镜头计划模板改为三层轨迹；角色/场景识别复用现有定妆卡（cast/scene 分类） |
| G2 | 视频→下一镜头视频的参考（过渡视频/尾帧锚） | video 节点只支持「图节点 → 首帧」引用；video→video 边无运行时语义 | **尾帧提取适配器**：边 video→video 时，runner 自动抽前一视频尾帧作为下一镜头首帧参考（i2v 模型都吃图，不需要新模型能力）；过渡视频留二期 |
| G3 | VLM best-of-k 质检 | 单次生成，人眼挑 | 节点级「生成 3 选 1」选项（**花 k 倍额度，需用户拍板默认关**） |
| G4 | 结构化镜头规格进提示词（机位/运镜/时长） | prompt 自由文本 | 拆镜头 prompt 模板补镜头语言字段；与现有 archetype 参数体系合流 |
| G5 | 层级分解 + RAG（百镜头级长片） | 创作文档单层拆镜 | 暂缓——Nomi 当前目标是 6-20 镜头短片，单层够用 |

## 四、落地方案对比（R3）

| 方案 | 用户看到什么 | 代价 |
|---|---|---|
| **A 轨迹规划（推荐，G1+G4）** | 拆镜头后画布直接出现三层轨迹：定妆参考卡 → 关键帧 → 视频节点，连好边、配好模型；点「批量生成」按波次自动跑（参考先生成已支持） | 改 agent prompt/计划模板 + 计划卡分层展示；~2-3 天工作量；不新增运行时机制 |
| **B 尾帧接力（G2，可选能力非默认）** | `video→video` 连边才生效：生成下游视频时自动用上游尾帧当首帧。**Agent 默认不连这种边**，只在镜头为连续动作（同场景顺接）时在计划里建议；场景切换/跳剪不连。用户随时手动连/删边 | runner 加尾帧抽取（ffmpeg 抽末帧已有导出基建可复用）+ 边语义扩展；~1-2 天 |
| C best-of-k（G3） | 节点参数里加「生成 3 张挑 1」开关 + VLM 自动评分排序 | 3 倍生成额度 + VLM 判分额度（用户资源）；判分质量依赖 VLM |
| D 全套 ViMax 管线复刻 | 一键从想法到成片的离线管线 | 与画布交互哲学冲突（用户失去掌控点）、工程量大、违背本地交互定位 |

**推荐组合：A 为主，B 作为可选边语义一并实现**（能力在、默认不强加：用户拍板 2026-06-12「不要卡死 B，万一用户不需要」）；C 二期可选（额度拍板）；D 不做（我们的画布即轨迹，比黑盒管线强）。

## 五、一期实现要点（A+B，拍板后细化为执行文档）

1. **计划模板**（`generationCanvasAgentClient.ts` system prompt + storyboard skill）：拆镜头输出三层节点 + 跨层边；角色/场景去重（同一角色一张定妆卡，多镜头共享引用）。
2. **计划卡分层渲染**（`AgentPlanCard`）：按 参考/关键帧/视频 分组展示，一次批准（已有原子机制 c493929）。
3. **尾帧适配器**（`runner/`）：`video → video` 边在 `buildDependencyWaves` 就绪后、提交生成前抽前序视频末帧，注入 referenceImage 槽；缺尾帧（前序未生成）时按现有「参考未就绪」拦截语义处理。
4. **布局**：三层轨迹按列分层排布（参考列 → 关键帧列 → 视频列），顺带修审计 bug D（网格避让）。
5. 验收：真实 E2E 一条 3 镜头轨迹跑通（接入即验证纪律）+ 对账/撤销/计划卡断言。
