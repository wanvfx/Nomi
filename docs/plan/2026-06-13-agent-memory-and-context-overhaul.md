## 0. 误解纠正 + 必须 informed 重决的事实

### 0.1 Agent 不是「看不到历史」

同一次运行内 Agent 看得到全部历史（含工具调用），创作区与画布区共享（`agentChatV2.ts:453/475/491`）。真问题是：① **重启即失忆**（喂模型的历史是内存 Map，`agentChatV2.ts:385`，退出即清；磁盘只存气泡 `{id,role,content}`，重启只回界面不回模型）；② **30 轮/24k token 上限**硬截（`agentChatHarness.ts:19-20`）。

### 0.2 这四件事 master-plan 都判过，3 件被刻意延后/砍掉

| 诉求 | master-plan | 当时决定 | 理由 |

| ① 重启不失忆 | S1/P-3 | 气泡落盘+分隔线**已做**；真 resume**挂触发器** | 真 resume 应「从 EventLog 重建，有损需声明」(:182) |

## 1. 铁律（不可破）

- **P1**：新东西沿 EventLog 现有缝生长，不造第二份真相源（master-plan §1.1：`agent.turn.finished` = 语义真相，气泡 store = 展示缓存，:85）。
- **不许界面说谎**（master-plan §5）：UI 呈现的「AI 记得的范围」必须 ⊆ LLM 实际范围，宁少不多。**假透明比不透明更糟**——这条直接判了「有损 resume 却让 AI 装作记得」是违规。
- **精度即内容（v2 新增铁律）**：精确约束（色值如 `#8B0000`、尺寸、型号、数字、锁定值）在任何记忆/压缩管线里**必须逐字保留，禁入摘要**。摘要只可损失「我们聊过镜头节奏」这类软脉络，绝不可把「暗红 #8B0000」压成「红色」。

## 2. 战线 A — 重启不失忆【核心拍板，评审重写】

### 评审打穿的事：v1 推荐的「路 A」做不到

v1 推荐「从 EventLog 的 `agent.turn.finished` + `agent.tool.*` 重建 CoreMessage 喂模型」，号称 P1 安全且「有损但够用」。**两路评审一致证伪**：

- EventLog 为了 4KB/截断纪律，**存的根本不够重建一份能喂回模型的对话**：user 消息只存头 **256 字**（`agentChatTrace.ts:9,45`）、附件**完全不进事件**、assistant 文本截 **2KB**、tool 结果是 **2048 字 JSON 串**（parse 不回对象，`agentChatTrace.ts:71`）、tool args 超 4KB 落 sidecar 残值、且 `tool.proposed/completed` **不带 proposalId**，分不清哪个 tool-call 模型真收到了、哪个是被用户拒绝的（重建必产**孤儿 tool pair**，provider 直接 400）。
- 降级方案（丢掉全部 tool 往返、只拼纯文本）→ 「记得要点」缩水成「记得每轮前 2KB 的几句话、user 只剩 256 字、附件全丢」。这远不是「摘要级」，是**残片级**。
- 要让 EventLog 够重建，就得把 user 全文/tool 结构化全文塞进事件——**那正是路 A 自己反对的第二真相源**。**「P1 安全 + 够用」的路 A 不存在。**

### 所以真正的选择是这三个（不是 v1 的两个）

| | 选项① 残片级重建 | 选项② 持久化模型工作缓存（业界主流） | 选项③ 不 resume 对话，只靠记忆卡 |
|---|---|---|---|
| 做法 | 从 EventLog 拼纯文本对话（tool 往返/附件/精度全丢） | 把内存 Map 的 `CoreMessage[]`（含 tool parts）落盘，重启读回 Map（Cline 模式） | 维持现状（重启=新会话），但把记忆卡（C）做到能逐字保留精确设定 |
| resume 质量 | 残片级，且静默失真（违「不许说谎」） | **逐字无损** | 不恢复对话，但恢复全部持久事实 |
| P1 | 纯（但没用） | **重定性争议**：这是持久化「模型工作缓存」（第三种东西），还是 EventLog 之外的「第二对话真相」？见下 | 纯，且正是 master-plan 原意 |
| 代价 | 名不副实 | 落盘一份完整对话（要解决「它和 turn.finished 谁是真相」）；JSON 易损→建议 SQLite | 对话本身不回来，用户可能仍觉得「它忘了我们聊的」 |

### v2 对 P1 的重定性（关键论点，请你裁决）

master-plan「日志=语义真相、store=展示缓存」这条，原本针对的是**UI 气泡**。但**模型的工作记忆（那个内存 Map）是第三种东西**——既不是审计真相，也不是 UI 展示，而是**喂给模型的工作缓冲**。把它落盘跨重启续用（选项②），更像「把进程的 RAM 存到磁盘再恢复」，而非「另立一套和日志打架的对话真相」。日志依旧是审计/记忆提炼的语义真相，落盘的 CoreMessage 只是续命工作缓存、允许有损允许与日志不同（日志本来就只存 head）。

**这条重定性成立不成立，是本方案最核心的一次拍板**——成立则选②（业界做法、逐字续聊）；不成立（坚持任何完整对话落盘都算第二真相源）则只能选③（不 resume 对话，全压到记忆卡上）。选项①评审判定名不副实，淘汰。

### 不论选②还是③，精度铁律都要先落

评审最尖锐的一条：`#8B0000 暗红 → 红色` 的静默失真。**精确约束必须逐字进记忆卡、永不摘要**（§1 铁律）。现在的零 LLM 规则提炼器能不能抓到色值、抓进去丢不丢精度，**没验证过**——这是选②/③共同的前置功课。

### 用户看到的变化

- 选②：重开项目，接着上次的话题问，AI 逐字记得。分隔线从「已不再记得」改成无（或「已恢复」）。
- 选③：重开是新对话，但 AI 仍记得全部角色/约束/精确设定（记忆卡），不会问你已经定过的事。

## 3. 战线 B — 长对话压缩【评审：与 A 是同一个东西，且挂触发器别提前】

### 评审打穿的事

1. **A 和 B 操作的是同一个对象**——「喂模型的 `CoreMessage[]` 历史视图」。v1 让 A 走「EventLog→投影」、B 走「内存→capAgentHistory」两条独立管线，代码零复用，导致 B 在内存里压出的摘要**重启后 A 读不到 → 丢失**。必须收口成**一份历史视图函数**：`运行时内存 ∪ 冷启动恢复 → 统一压缩/摘要 → 发送`。
2. **现状已有滑窗+载荷瘦身，只缺「摘要」那一环**（`agentChatHarness.ts` 自承 later step）。但 `context.compacted` 事件的 summary >512 字会被 EventLog 截成 sidecar = 换名的第二真相源；摘要塞 system role 会破坏 Anthropic 缓存断点（`agentChatHarness.ts:135`）——要塞就塞成 user/assistant 文本。
3. **触发器数据此刻就在落盘**：`context.capped` 事件一直在记（`agentChatV2.ts:494`）。master-plan 立的「`capped ≥3次/周` 才做压缩」闸门，**先去读这个项目真实 capped 频次**，没到阈值就别提前做 B。

### 结论：B 维持挂触发器，先不做。先把 A（选②或③）和 C 做好，A 的工作缓存若落盘，长度问题大半自然缓解；真撞上限再做摘要，且与 A 共用一份历史视图函数。

## 4. 战线 C — 记忆卡增强【手动加保留；软事实必须改提议态】

### C1 用户手动加记忆——保留，但 v1 把它当「纯扩」是错的

评审发现：`getProjectMemory` 回放循环**根本没有 `memory.fact.added` 分支**（只特判 removed/corrected，其余 `memory.*` 被跳过，`projectMemory.ts:174`）——「仿 removeMemoryFact」这句错了，手动 fact 会进不来，必须**新增回放分支**。且手动 fact 的 id 必须用**独立 `user:` 前缀**与规则键（`node:`/`lock:`/`pref:`）隔离，否则共享扁平 id 空间 + 墓碑表会串味（自动提炼复活用户删过的 fact）。还有并发：`getProjectMemory` 读改写非原子，创作区+画布区同时挂载会重复写回执污染时间线——**必须 per-project 串行化**。仍做，但按这些约束实现。

### C2 LLM 软事实——评审：必须改成「提议态」，否则冒犯+事故双杀

v1 说软事实走「同一 upsert + 三不变量」自动进记忆——**评审判定这恰恰错了**。创作者对「被机器揣测品味」容忍度极低；自动悟出「偏好冷色调」并**静默注入**下次生成，会把本该暖色的镜头带跑 = 冒犯 + 事故。

**必改**：软事实不复用硬事实的自动 upsert 路径，而是**提议态**——AI 推断的事实先作为「建议」显示（带「AI 推断」角标），**用户点头才转正进记忆卡**（复用 harness L 层确认门的灵魂）。默认仍关，做也要做成提议。异步旁路的输入源/失败降级/并发锁评审也点了名（`chooseTextModel` 无 key 直接 throw，要静默降级）。

### C3 相关性检索注入——不做（评审认同）。记忆量小时全量比 RAG 准且省。

## 5. 战线 D — 记忆历史时间线【评审：必砍独立弹层，降级为单条来源提示】

评审（PM+用户）最尖锐的判决：**v1 给 D 排第一的理由「底层数据现成、零新结构」是把「实现成本低」误当「用户价值高」——这俩毫无关系**。视频创作者赶片子时不会去翻「这条‘主角穿红裙’3 天前从镜头 2 自动记的」。master-plan 自砍「无人要、自嗨」**依然成立**。

且评审发现「纯消费现有事件」也不是零逻辑：`memory.fact.removed` 有 user-removed / source-gone 两种 reason，`corrected` 有 user/auto 两种 origin，时间线不区分就会**撒谎**（把系统自动删显示成用户操作）。

**必改**：砍掉独立时间线弹层。真需求「这条记忆哪来的、能不能改」由**记忆卡单条的「ⓘ 来源」悬浮提示**覆盖即可（一条 fact 的 sourceSeqs 已有数据），不做全景时间线。

## 6. UI 评审必改（前端+设计）

- **参照件纠错**：要仿的弹层是 `AssetPickerPopover.tsx`（BodyPortal+fixed），**不是不存在的 SettingsPopover**。对话面板 aside 是 `overflow-hidden`（`CanvasAssistantPanel.tsx:467`），任何 absolute 弹层会被裁——弹层强制走 portal+fixed。
- **折叠态零操作**：`MemoryFold` 折叠条一个字不加（维持与孪生的 `AssistantToolsFold` 视觉对称）；「+添加」「来源」入口**只在展开态**出现，复用现有 `text-micro` 灰字 span 和 inline input（零新组件零新 token）。
- **emoji 违规**：🕘/💡 等一律换 `@tabler/icons-react`（`IconHistory`/`IconClock`，对齐现有 `IconBrain size=13`）。
- **三态分隔线收口**：现「已不再记得」（`staleConversationDivider.tsx`）与新「已压缩为摘要」「已恢复」外观相同、语义相反，会同屏打架。**泛化成一个 `ConversationBoundaryDivider({variant})`**，三态共用布局、加前导图标区分、按信息丢失程度给不同视觉权重。
- **必出样张（R8）**：① 记忆折叠条展开态新布局（双入口+「AI 推断」角标行）；② 三态分隔线同屏最坏情况；③（若 D 不砍）记忆历史列表。

## 8. 排期【评审反转：止血优先于建信心】

v1 排期（D→C1→A→B）被判「把唯一真痛点 A 排第 3、把判过死刑的 D 排第 1 = 工程偷懒不是产品逻辑」。v2 反转：

1. **S-A 重启 resume（选②或③）+ 精度铁律**——唯一真痛点，先止血。
2. **S-C1 手动加记忆**——便宜的真需求。
3. **D 降级**为单条记忆「ⓘ 来源」悬浮（不做时间线弹层）。
4. **B 维持挂触发器**——先读 `context.capped` 频次，真撞上限再做，与 A 共用历史视图。
5. **S-C2 软事实**——若做，提议态，最后做。

每片独立 commit，五门 + R13 走查 + R8 样张（UI 项）。
