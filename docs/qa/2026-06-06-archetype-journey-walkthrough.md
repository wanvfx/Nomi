## 体感对的（保留）
- **生成方式分段切换**：active 态明显，切它正下方参考槽**当场变化**（防遮挡设计成立，C2b 目标达成）。
- **统一意图词 + vendor 副标签**：「单图首帧」配「该模型称『首帧』」——黑话感降下来了，U1 落地。
- **全能参考的角色图**：① 数字徽标 + 组下「顺序对应 prompt 里的 character1…9」+ prompt 旁 accent 提示，
  槽→prompt 词的链接很显性，U2 落地得好。
- **跨模型一致**：HappyHorse 4 模式合 1、i2v 自动不显示比例（U3），与 Seedance 同位同名——体感是「同一种工具」。
- 项目库第一印象干净，主操作（新建/示例）清晰。

## 问题（按疼痛度排序 → 转优化项）

### 🔴 P1
**J1 底部参数全是「裸值无标签」，新用户读不懂。**
`720p / 16:9 / 5 / 开启`、`1080p / 自动 / 随机`——这些下拉只显值不显名：「5」是什么（秒？）、「开启」开的什么、
「随机」是什么？(`valueOnly` 把 label 设了 `sr-only`。) 样张 v3 本来设计了**带标签的设置弹层 + 底部摘要芯片**，
但实现期延后了，于是现状把一排无标签裸值直接铺在节点底。→ 优化：按样张做设置弹层（标量参数带标签往下弹）+
芯片摘要；至少给底栏控件补可见 label / title。**（规则 8：先出样张再改。）**

**J2「视频编辑」模式的「源视频」槽缺失 → 该模式实际不可用。**
HappyHorse video-edit 的 hint 写「源视频 + 0–5 张参考图」，但 UI 只渲染了「参考图」数组槽，**没有源视频输入**
——因为 `source_video` 这个 slot kind 在 `archetypeModeSlots`（只管首/尾帧）和 `archetypeModeArraySlots`
（只管 image/video/audio_ref）里都没接 UI（plan §1 明确「video-edit 源视频 UI 本轮不重点」）。但对用户来说，
这是个**点进去填不了东西的死模式**。→ 优化二选一：① 补 `source_video` 上传槽；② 在它能用之前先把该模式从分段里
隐藏（不给用户一个用不了的入口）。

### 🟡 P2
**J3 模型名截断**：选择器显示「Seedan」「HappyH」——宽度不够，看着像 bug。→ 加宽/换 title 省略。
**J4 空间偏空 / 空组常驻**：单图模式下「+」槽↔prompt↔底栏之间留白偏大，节点偏高；全能参考里**空的
「参考视频」「参考音频」组也常驻**占高度（样张 U3 早写过「空组未用时收成灰『+』」）。→ 压缩留白 + 空数组组收成占位。

### 🟢 P3（文案/细节）
**J5** video-edit 的「参考图」组复用了「顺序对应 character1…9」说明 + 触发了 character promptCue，但 edit 的参考图
未必是 character 语义——文案对该模式略不准（`numbered`/`caption` 对所有 image_ref 一刀切）。
**J6** 单图「+」槽无可见标签（靠上方 hint 隐含），与 chip 关联不够显性。
**J7** 跨模型保留参考图（拍板的设计）会把 Seedance omni 放的角色图带进 HappyHorse video-edit 的参考图槽——
行为符合「切模型不丢图」意图，但落到 edit 槽略意外；可接受，记录备查。

## 结论
档案层的核心体感（模式切换、统一词汇、角色图 U2、跨模型一致）**立住了**。最该补的是 **J1（参数无标签，做样张里的设置弹层）**
和 **J2（video-edit 死模式）**——前者是高频体感伤，后者是功能空洞。二者都是用户可见改动，按规则 8（样张）+ 规则 7（评审）走，
不擅自改。真实生成验证仍待用户额度（`tests/ux/seedance.e2e.mjs` / `happyhorse.e2e.mjs`，KIE_API_KEY 门控）。
