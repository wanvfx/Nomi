## 0. 速读 (Executive Summary)

**一句话定位**

> Nomi 是一个**本地优先、开源、AI-native** 的创作 IDE — 让一个人用一台电脑、一个项目文件夹，完成从剧本到成片、从构图到素材、从灵感到导出的完整生产链。

**我们的产品形态正在从"AI 视频画布"演化为"AI x 设计一体的创作 OS"。** 与 Figma 的关系不是竞争而是吸收 — 把 Figma 最底层的几何、文本、对齐、图层等基础设计能力放进 AI 画布里，让设计原语和 AI 生成原语在同一张画布上平等存在。

**核心信念**

1. 生成模型会快速同质化，护城河不在"谁生成得更好"，而在**"谁能把生成纳入完整生产链"**。
2. 创作者今天在 5 个工具里来回切换（脚本 + 生图 + 生视频 + 设计排版 + 剪辑），痛点是**工具切换的认知摩擦**和**资产在多工具间漂泊**。
3. 桌面 + 本地项目 + 开源扩展 = 可信生产力工具应该有的底座。SaaS 无法满足"作品和资产真正属于我"的诉求。
4. AI 不是工具栏里的一个按钮，AI 是**画布上的一种原语** — 和矩形、文本、图片一样平等。

**12 个月战略路径**

| 阶段 | 时间 | 主题 | 里程碑版本 |
|---|---|---|---|
| Phase 1 | 0–3 月 | 本地生产 IDE 基础 | v0.2 – v0.4 |
| Phase 2 | 3–6 月 | 视觉设计层 + AI 融合 | v0.5 – v0.7 |
| Phase 3 | 6–12 月 | 生态化、社区化、半商业化 | v0.8 – v1.0 |

## 1. 战略定位

### 1.1 重新定义产品类目

**Nomi 不是另一个 AI 视频工具，而是 AI 时代的创作 IDE。**

"IDE" 的类比是刻意的：
- VSCode 之于程序员 = Nomi 之于 AI 创作者
- 一个本地项目就是一个文件夹（vs SaaS 项目=远端记录）
- 扩展生态决定上限（vs 平台功能等待官方）
- 工作流可复现、可分享、可 fork（vs 黑盒云端工作流）
- 命令行 / 自动化能力对高级用户开放（vs 仅 GUI）

### 1.2 三句话回答"为什么是我们"

> **桌面端给我控制权 — 所有源文件、素材、模型配置、生成历史都在我电脑里。**
>
> **开源给我安全感 — 我能审计代码、定制流程、永远不被锁死。**
>
> **设计 + AI 一体给我顺滑感 — 我画布上能同时有矩形、文本、生成的图、生成的视频，它们是平等的原语。**

### 1.3 反向定义：我们绝不做什么

| 不做 | 原因 |
|---|---|
| 云端协作 IDE | SaaS 资本战，无法对抗 Figma/Adobe |
| 模型自研 | 烧钱无尽头，下游应用比上游模型早死 |
| 影视工业级 SaaS | Laper 的主场，资源不够 |
| 社交化创作社区 | TapNow 的主场，重运营 |
| 中心化模板市场 | 早期生态太小，先做本地 pack 即可 |
| 移动端 App | 创作场景重，移动是消费端 |

### 1.4 升级后的差异化矩阵

| 维度 | Figma | Midjourney/MJ | Krea | Runway | ComfyUI | LibTV/TapNow | **Nomi** |
|---|---|---|---|---|---|---|---|
| 类目 | 设计工具 | AI 图像 | AI 创作画布 | AI 视频 | 节点图像引擎 | AI 视频画布 | **AI 创作 IDE** |
| 数据归属 | 平台 | 平台 | 平台 | 平台 | 本地 | 平台 | **本地** |
| 开源 | ✗ | ✗ | ✗ | ✗ | ✓ | 部分 | **✓** |
| 桌面端 | ✗(Electron 但本质 Web) | ✗ | ✗ | ✗ | ✓ | ✗ | **✓** |
| 设计原语 | **★★★★★** | ✗ | 简单 | ✗ | ✗ | 简单 | **★★★★** (目标) |
| AI 原语 | 插件 | **★★★★★** | **★★★★★** | **★★★★★** | **★★★★★** | **★★★★** | **★★★★** |
| 长项目记忆 | ★★★ | ✗ | ✗ | ✗ | ★ (手动) | ★★ | **★★★★★** |
| Agent 自动化 | ✗ | ✗ | ✗ | 简单 | ✗ | ★★ | **★★★★** (目标) |
| 任意模型 | ✗ | ✗ | ✗ | ✗ | ★★★★★ | ★★ | **★★★★★** |
| 学习曲线 | 中 | 低 | 低 | 低 | 极高 | 低 | **中** |

**Nomi 的独特坐标**：在表格中**只有 Nomi 能在 5 个关键维度上同时打钩** — 本地、开源、桌面、长项目记忆、任意模型。

## 2. 用户与场景

### 2.1 第一优先级用户（核心 ICP）

**"独立 AI 创作者"** — 一个人完成从故事到成片的整个流程。

**典型画像 A：漫剧/短剧创作者（小苏）**

- 25 岁，二次元/漫剧爱好者，业余创作 AI 漫剧短片在 B 站/抖音发布。
- 当前工作流：在 Word 写剧本 → MJ 生图 → Runway 生视频 → 剪映剪辑 → Photoshop 调封面 → 发布。
- **痛点排序**：
  1. **5 个工具的资产在硬盘里散落**，命名混乱，找不到之前生成的角色参考图。
  2. **角色一致性灾难** — 每个镜头都要重新写一遍角色描述，仍然变形。
  3. **生成失败重试痛苦** — 云端 30 张图跑一半失败，要从头开始。
  4. **导出格式繁琐** — 抖音要 9:16、B 站要 16:9，每次重新排版。

**典型画像 B：内容博主/UP 主（老王）**

- 35 岁，做"X 用 AI 做 Y"类教程视频。
- 需要：快速做 demo、抓取流程截图、做对比图。
- **痛点**：
  1. AI 工具种类太多，每个都要学怎么接 API。
  2. 没法把"一个完整工作流"完整地展示给观众。
  3. 想把自己摸索出来的 prompt + 参数分享给观众，但没载体。

**典型画像 C：独立游戏/动漫工作室美术总监**

- 30 岁，2–5 人小团队，做独立游戏概念设计或动画样片。
- 需要：批量生成角色立绘、场景参考、storyboard，**保密性高**。
- **痛点**：
  1. 不敢把未发布 IP 上传到云端工具。
  2. ComfyUI 太工程化，团队美术受不了。
  3. 想要 Figma 式的协作但同时要 AI 生成能力。

### 2.2 第二优先级用户

**"AI 工作流极客 / 开源贡献者"**

- 喜欢折腾各种模型，写脚本调 API，对开源有热情。
- Nomi 给他们的价值：**一个能让他们打包并分享自己模型工作流的产品壳**。一个人调通的 prompt + 节点图 + 模型配置可以做成 Recipe，发布到社区，被几千人复用。

### 2.3 暂不优先

- 大企业制片协作（Laper 主场）
- 营销视觉电商团队（TapNow 主场）
- 纯专业影视后期（DaVinci/Premiere 主场）

### 2.4 典型用户旅程（小苏案例）

> **当前**: 早上 9 点开始，到晚上 12 点出 1 集 5 分钟漫剧。中途切换 ≥ 30 次工具。
>
> **Nomi 后**: 早上 9 点新建 Nomi 项目 → 写剧本 → Agent 自动拆 12 个镜头 → 批量生成图 → 拖入时间轴 → 加字幕 → 导出 9:16 和 16:9 双版本 → 项目打包归档。
>
> **关键指标**：完成时间从 15h → 4h；工具切换次数 30+ → 0；资产丢失率 ~10% → 0%。

## 3. 竞争分析（基于 v1 PRD 扩展）

### 3.1 直接竞品（已在 v1 PRD 详述）

参考 `nomi-differentiation-prd-2026-05-23.md` §3。简要复述结论：

- **Laper**：影视前期 SaaS。我们不抢专业制片，但学习它的"项目智能"概念。
- **TapNow**：在线创意画布。我们不抢社交，但学习它的"AI 作为原语"心智。
- **LibTV**：Web AI 视频画布。我们的桌面 + 本地 + 开源是它无法做的根本差异。

### 3.2 间接竞品（v1 PRD 漏掉的）

**Figma + AI Plugins（最大威胁）**

- Figma 是设计工具霸主，已经有 AI 插件生态（Magician、Diagram、Genius）。
- **威胁**：Figma 可能整合 AI 生成作为原生功能，吃掉"设计 + AI"市场。
- **应对**：Nomi 押注"视频和长项目"这个 Figma 弱区。Figma 是静态设计单文件思维，Nomi 是动态项目+时间轴+ Agent 自动化思维。**视频是 Figma 进不来的方向**。

**Krea AI（最相似的产品哲学）**

- Krea 是在线版的"实时 AI + 简单画布"，主打 AI 生成时的视觉反馈和迭代。
- **差异**：Krea 是 SaaS、不开源、无项目记忆、不做视频时间轴、不做 Agent。
- **应对**：Nomi 是 Krea 的桌面 + 开源 + 长项目记忆版。差异够大。

**Genmo / Pika / Luma（视频生成模型方）**

- 它们是生成能力提供方，不是工作流产品。Nomi 应该**作为它们的下游消费方**，把它们当作可接入的 provider。
- **应对**：Nomi Provider Pack 体系让用户 5 分钟接入它们任何一家。

**ComfyUI（开源同盟，不是对手）**

- ComfyUI 极度工程化，是底层节点引擎。Nomi 是上层创作 IDE。
- **可能性**：Nomi 通过 Provider Pack 调用本地 ComfyUI 工作流，把 ComfyUI 工程能力暴露给非工程师创作者。**这是双赢的协同**。

**Adobe Firefly + Creative Cloud**

- 资源最雄厚的对手。但 Adobe 路线是"现有工具加 AI 模块"，难以做出 AI-native 的范式革新。
- **应对**：Nomi 直接做 AI-native，不背任何历史包袱。

### 3.3 总取舍

| 我们押注 | 我们放弃 |
|---|---|
| 桌面 + 本地 + 开源 | 云端协作 SaaS |
| AI x 设计的画布融合 | 纯生成或纯设计 |
| 视频 + 长项目记忆 | 静态单文件设计 |
| 任意模型接入 | 自研模型 |
| 个体创作者 + 小工作室 | 大企业 + 影视工业 |
| 单机生产力 → 可选云同步 | 强协作 → 弱单机 |

## 4. 产品差异化的四根支柱

支柱 1–3 来自 v1 PRD，支柱 4 是 v2 新增。

### 支柱 1：本地项目仓库（已有，需深化）

- `Nomi Projects/` 是真实文件夹，可备份、迁移、Git 化、发给别人。
- 项目包 `.nomiproj` 跨机器可复现。
- 详见 v1 PRD §4.1。

### 支柱 2：开源扩展生态（已有，需深化）

- Provider Pack：模型接入包。
- Skill Pack：Agent 能力扩展。
- Workflow Recipe：完整工作流模板。
- 详见 v1 PRD §4.2。

### 支柱 3：三层生产系统：Memory + Graph + Agent（已有）

- Project Memory Layer：剧本、角色、场景、镜头、资产、导出、模型证据。
- Production Graph Layer：画布 DAG + 时间轴 = 可复现可审计生产图。
- Agent Runtime Layer：读取状态 → 规划 → 工具调用 → 写回 → 日志。
- 详见 v1 PRD §4.3。

### 支柱 4：视觉设计层（NEW — 本次核心新增）

**核心命题**：

> AI 创作工具的下一个进化方向，不是更强的生成模型，而是**让 AI 原语和设计原语在同一张画布上平等存在**。

**为什么这个方向是对的**：

1. **文本提示的带宽极限**：纯文本 prompt 表达力有限。Krea/Magnific 已经证明"用视觉引导 AI 生成"是降低门槛的关键。
2. **生成后的二次处理需求**：生成结果几乎总是需要剪裁、调整、组合、加文字。当前必须切到 PS/Figma。
3. **构图意图的视觉表达**：创作者头脑中的构图本身就是视觉的，强迫他们翻译成文字是认知摩擦。
4. **多素材组合是常态**：故事板、海报、社交卡片本质都是多 AI 生成素材 + 文字 + 几何形状的组合。
5. **AI 涂改与局部编辑**：inpaint / outpaint / 局部 regen 都需要在原图上画选区，这是设计工具天然能力。

**Nomi 视觉设计层的 5 类基础能力**

| 类别 | 具体能力 | AI 化学反应 |
|---|---|---|
| **几何原语** | 矩形、椭圆、线、箭头、钢笔路径 | 路径作为 ControlNet sketch；选区作为 inpaint mask |
| **文本原语** | 文字图层、字体、颜色、对齐、行距 | 文字内容自动作为生成 prompt 上下文；文字层 + 图层组合直接导出社交卡片 |
| **画框 (Frame)** | 输出尺寸预设（9:16 / 16:9 / 1:1 / 4:5 / 自定义） | 画框定义生成尺寸；画框间可继承风格（"和 Frame A 同风格"） |
| **图层与组** | 图层面板、可见性、锁定、组合、z-order | 多个生成结果 + 设计原语组合后整体导出；图层是"AI 生成证据"的载体 |
| **对齐与变换** | 8 向对齐、分布、群组移动旋转缩放、对齐网格、智能引导线 | 多个生成 frame 自动 grid 化布局；对齐生成结果做 storyboard |

**关键原则：AI 是原语，不是按钮**

举例对比：

| 当前竞品的范式 | Nomi 的范式 |
|---|---|
| 点 "Generate" 按钮 → 弹窗输入 prompt → 看结果 | 拖一个 "AI 生成区"（像拖一个矩形）→ 在区域内打字 prompt → 实时生成 |
| 在 PS 里画完构图 → 截图 → 上传到 MJ → 描述构图 | 在 Nomi 画布上随手画几个矩形和箭头 → 选中 → "用这个构图生成" |
| 生成完图片 → 下载 → 在 Figma 里加文字 | 生成完图片就在画布上 → 拖个文字图层叠上去 → 直接导出 |
| 想要 inpaint：切到 SD WebUI → 涂遮罩 → 输入 prompt | 在画布上的图上画一个矩形/钢笔路径 → "用这个区域替换为 X" |

**v2 视觉设计层的产品坐标**

- 不是 Figma：我们不做组件库、Auto Layout、设计系统级能力。
- 不是 PS：我们不做像素级编辑、滤镜、调色。
- 我们是 **"AI-native 的 Figma + Krea 子集"**：刚好够用、AI 优先、专为创作者。

## 5. 视觉设计层功能详细规划

### 5.1 v0.5 视觉编辑基础（Phase 2 启动）

**P0 工具栏**：

- 选择工具（默认）、矩形、椭圆、直线、箭头、文字
- 钢笔/路径（用于 inpaint mask）
- 取色器
- 注释 / 便签

**P0 画布交互**：

- 多选（rubber band + shift-click）
- 群组 / 解组（Cmd+G / Cmd+Shift+G）
- 8 向对齐 + 分布
- 智能引导线（snap to neighbor）
- 网格 + 像素吸附
- 复制粘贴含样式
- 撤销/重做（已有）

**P0 图层面板**：

- z-order 拖拽
- 可见性 / 锁定 / 重命名
- 群组层级树

**P0 画框 (Frame)**：

- 拖动创建 Frame，预设尺寸：9:16 / 16:9 / 1:1 / 4:5 / 自定义
- 内部内容裁剪
- 可作为导出单元（点击 Export Frame）

### 5.2 v0.6 AI x 设计的化学反应

**"生成原语"作为一种图层**：

- 拖一个"AI 生成 Frame" → 输入 prompt → 在该 Frame 内生成
- 同一画布上可以有多个生成 Frame，互不干扰
- 生成结果是图层，可以叠加其他设计元素

**Sketch-to-Image**：

- 用矩形/椭圆/线条画粗略构图 → 选中 → 右键 "Generate from sketch"
- 后端调用 ControlNet sketch/depth 模型
- 草图保留在画布作为"意图证据"

**Region Inpaint**：

- 在生成结果上画矩形或钢笔路径 → 右键 "Inpaint this region"
- 路径自动转 mask
- 仅替换选区内容

**Style Reference Drop**：

- 把任何图片拖到一个 Frame 旁 → 自动标记为该 Frame 的 style reference
- 生成时 extras 自动带上

**Color Palette from Selection**：

- 选中一组元素 → 提取主色调
- 主色调作为 prompt 上下文（"用以下颜色：#A1B2C3..."）

**Multi-Frame Variants**：

- 选中一个 Frame → "生成 4 个变体" → 自动 grid 化排列 4 个新 Frame
- 用户挑选满意的，删掉其余

### 5.3 v0.7 设计 + 故事板融合

**Storyboard Frame Sequence**：

- 把多个 Frame 按顺序连接 → 形成故事板序列
- 可一键转为时间轴片段
- 镜头之间的转场用画布上的箭头表达

**Annotation Layer**：

- 在 Frame 上加 sticky note / 旁白文本
- 注释不进入生成 prompt（除非显式标记）
- 导出 PDF 故事板时注释作为镜头说明

**Component-lite（轻组件）**：

- 把一组元素保存为"角色卡片"或"场景模板"
- 在新 Frame 中插入卡片，自动作为 character reference

### 5.4 暂不做（即使 Figma 有）

- 真正的 Auto Layout（响应式约束）
- Design Tokens / 变量系统
- 多人实时协同编辑
- 完整组件库 / Design System
- Boolean 路径运算（union/subtract/intersect）
- 矢量编辑器级别的钢笔
- 像素级编辑

这些等到 v1.0+ 看用户是否需要再加。MVP 不追求 Figma 等价。

## 6. 12 个月路线图

### Phase 1: 本地生产 IDE 基础（0–3 月，v0.2 – v0.4）

**目标**：完成 v1 PRD §6.1 的闭环 — 从剧本到 MP4 在本地完整跑通。

| 版本 | 时间 | 核心 |
|---|---|---|
| v0.2 | M1 | 项目包 1.0：标准化目录、健康检查、打包/导入 `.nomiproj` |
| v0.3 | M2 | Provider Pack：文档→草案 Agent、三类 API 形态模板、测试 UI |
| v0.4 | M3 | Agent 可执行链：plan schema、工具权限、run log；桌面 ffmpeg 导出 |

**Phase 1 验收**：

- 用户能从 0 安装到导出第一个完整 MP4 < 60 分钟
- Star ≥ 500，活跃用户 ≥ 50

### Phase 2: 视觉设计层 + AI 融合（3–6 月，v0.5 – v0.7）

**目标**：让 Nomi 成为**唯一一个把 Figma 式编辑和 AI 生成在画布上平等融合的桌面工具**。

| 版本 | 时间 | 核心 |
|---|---|---|
| v0.5 | M4 | 视觉编辑基础：几何/文本/Frame/图层/对齐 |
| v0.6 | M5 | AI 化学反应：sketch-to-image、region inpaint、style ref、palette、variants |
| v0.7 | M6 | 故事板融合：Frame 序列→时间轴、annotation、轻组件 |

**Phase 2 验收**：

- 用户能完全在 Nomi 内完成一张 storyboard，不切到 Figma/PS
- 至少 3 个 Provider Pack 支持 ControlNet/Inpaint
- Star ≥ 2000，活跃用户 ≥ 500

### Phase 3: 长片基础设施（6–9 月，v0.8）

**目标**：让 Nomi 从"30 镜头短片工具"升级到"100-300 镜头中片可用"。本阶段所有功能由 `nomi-long-form-scale-audit-2026-05-24.md` 的 P0/P1 优先级驱动。

| 版本 | 时间 | 核心 |
|---|---|---|
| v0.8 | M7-9 | 资产库视图（脱离画布）+ 跨项目资产复用 + 项目级 git-like 分支 + 持久化任务队列升级 |

**Phase 3 验收**：

- 用户能管理 300+ 资产不卡
- 同一角色 / 场景能在 2 个项目里复用
- 批量生成 200 次可恢复（断点续跑）

### Phase 4: NLE 编辑器升级（9–12 月，v0.9）

**目标**：时间轴从"短片粗胚"升级为"30 分钟中片可用的多轨 NLE"。

| 版本 | 时间 | 核心 |
|---|---|---|
| v0.9 | M10-12 | 5 音轨（dialog/music/SFX/foley/ambient）+ 字幕系统（SRT 导入导出 + 多语言）+ 多 deliverable 工作流（同项目多 cut + 多分辨率派生）+ 后台渲染 |

**Phase 4 验收**：

- 用户能制作 30 分钟成片，含对白 / 音乐 / SFX 至少 3 轨
- 同项目可同时输出 16:9 / 9:16 / 1:1 三个 cut
- 后台渲染时编辑器可继续工作

### Phase 5: 生态化 + 长片完整闭环（12–18 月，v1.0）

**目标**：闭合长片场景；把 Nomi 从"个人工具"变成"生态化产品"。

| 版本 | 时间 | 核心 |
|---|---|---|
| v1.0 | M13-18 | 签名 + 公证发行；Pack / Recipe 社区索引；存储分层归档；长上下文 Agent (剧本分层 summarization)；色彩分级 (curves + LUT + 批量应用)；商业化前奏 |

**Phase 5 验收**：

- 社区 pack / recipe ≥ 30
- 至少一个非作者用户的 Recipe 在社区被复用 ≥ 100 次
- 可制作 60+ 分钟长片（端到端在 Nomi 内完成）
- Star ≥ 10000，付费意向用户 ≥ 200

### 路线图取舍说明

本路线图扩展自 v1（原 3 阶段），驱动文档：
- 当前 5 阶段对应 `nomi-long-form-scale-audit-2026-05-24.md` 12 需求群的 P0/P1/P2 分级
- 详细分级表见审计文档 §13 (新增章节)
- 短片用户（90%）在 Phase 2 结束即可享受端到端价值
- 中片 / 长片用户（10%）需等到 Phase 3-5，但他们是付费转化主力

### 7.6 商业化思路（远期，M9+）

**核心原则**：永远保留 100% 功能完整的开源免费版。商业化是"可选服务"。

**潜在商业化方向**：

1. **Nomi Cloud** — 可选项目云同步（不是必需，类似 1Password 的云端同步）
   - 个人版：免费 1GB / 付费 100GB
   - 团队版：项目共享、权限管理
   
2. **签名应用 + 自动更新** — 通过签名版安装包付费
   - 免费版自己处理签名警告
   - 付费版（$5-10/月）享受公证签名 + 自动更新
   
3. **Pack 商店** — 高级 Recipe / Provider Pack
   - 平台抽成 15-20%
   - 优质 Pack 作者分成收入
   
4. **企业版** — 企业内自部署 + 私有 Pack 仓库 + 优先支持
   - 年费 $5000+
   
5. **Token 中转 / 模型聚合** — 用户不想各处申请 key，付费用 Nomi 统一接入
   - 加价 10-20%
   - 这是低优先级，因为生态期价值有限

**何时商业化**：

- v0.x 阶段：完全不谈商业化，只谈"未来可能"
- v1.0 阶段：上线"赞助"按钮（Patreon / GitHub Sponsors）
- v1.x 阶段：上线第一个商业化产品（云同步 or 签名版）

## 8. 关键风险与对策

继承 v1 PRD §10 的 4 个风险，新增 4 个：

### 风险 5：视觉设计层和 AI 生成的范式融合实现成本高

问题：把 Figma 式编辑能力做进画布会让代码复杂度翻倍。

对策：
- 严格按 v0.5 → v0.7 分阶段推
- 每个能力都问"这个能力对 AI 化学反应有没有贡献" — 没有就砍
- 大量复用 React + zustand 现有架构

### 风险 6：Figma 推出原生 AI 视频功能

问题：Figma 资源充足，可能推出"Figma for video" + AI 一体方案。

对策：
- 我们的护城河是**桌面 + 本地 + 开源** — Figma 不可能放弃 SaaS 模式
- 提前 12 个月在视频 + 长项目方向建立心智，让 Figma 进来时已是 "Nomi vs Figma" 选择题
- 不和 Figma 拼通用，专攻 AI 视频创作者

### 风险 7：AI 模型 API 持续涨价 / 限制

问题：上游模型 API 涨价或限制，影响用户体验。

对策：
- 不绑定任何单一供应商
- Pack 体系让用户可以快速切换供应商
- 持续推进本地模型支持（Ollama、ComfyUI integration）
- 在 README 里直接告诉用户"你用谁的 key、你出钱、我们不抽水"

### 风险 8：开源社区冷启动 + Champion 培养失败

问题：早期没有种子用户和 Champion 就没有飞轮。

对策：
- 创始人自己每周做 demo（建立"用产品的人能做出酷东西"心智）
- 早期投入大量时间陪 10-20 个深度用户（一对一打磨）
- 不追求增长，先追求 NPS

## 12. 结论与下一步

**核心结论**：

Nomi 的差异化已经清晰 — **本地 + 开源 + AI-native + 设计与生成融合**。
v1 PRD 解决了"我们和别人不一样在哪"，v2 PRD 解决了"我们具体做什么、什么时候做、怎么让人知道"。

**接下来 30 天最关键的 3 件事**：

1. **v0.2 项目包系统上线** — 让用户真切感受到"本地项目"的差异化价值。
2. **完成 5 个官方 Provider Pack** — 用户接入第一个模型 < 5 分钟。
3. **建立内容节奏** — 每周 1 条 X demo，建立"持续在动的产品"心智。

**12 个月愿景**：

> 12 个月后，当一个 AI 创作者打开他的电脑，他会习惯性双击 Nomi 图标。
>
> 他的所有未发布作品、所有素材、所有 prompt 配方都在那个文件夹里。
> 他可以在画布上随手画几个矩形，输入一段 prompt，让 AI 在他指定的位置生成视频片段。
> 他可以让 Agent 帮他把一段文字拆成 12 个镜头，然后批量生成。
> 他可以把整个项目打包发给朋友，朋友打开就能继续编辑。
> 他可以从社区下载别人分享的 Recipe，瞬间获得别人 100 小时摸索出的工作流。
>
> 这个习惯，就是 Nomi 在 AI 创作时代的位置。
