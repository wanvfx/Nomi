## 一句话总结

本轮抓到的两篇新论文（HollyWood Town / OmniAgent，MUSE）和上轮头条 Co-Director **指向同一个结论，且更狠**：整个学界已经把「decompose-stitch 的级联误差」根因明确收敛到**「单向 forward 管线（DAG/瀑布）本身就是病——你必须有一个能回头改前面那步的闭环」**。HollyWood Town（2025-10，HKUST+Stanford+UChicago，含 Anyi Rao/Maneesh Agrawala，DirectorBench 同组人马）直接给出工程化解法：**从有向无环图（DAG）切换到「有限重试的有向有环图」**，让下游 agent（如发现某镜崩了）能把反馈回灌、**改前面阶段（剧本/分镜）再重展开**；MUSE（2026-02）把同一件事表述成 **plan-execute-verify-revise 闭环**，把叙事意图编译成对身份/构图/时序的「机器可执行约束」，生成后逐镜校验违反就修。这正是 Nomi 现在做不到的：拆镜头规划师产 storyboard 后，链路是**单向往下**，一镜崩了只能在那一镜局部 refine，**回不到「改前面分镜/剧本」那一步**。

本轮最该动的一件事：**给 Nomi 的拆镜头→生成链补一道「有限重试的回灌闸」——下游检测到镜级失败时，不只局部重生该镜，而是把失败反馈回灌给拆镜头规划师、允许改前面的分镜/剧本再重展开（bounded retry，比如最多 2 次防烧额度）。** 这是上轮 Co-Director「全局创作方向注入」之后的下一块拼图：上轮补的是「展开前先定统一方向」，本轮补的是「展开后发现错了能回头改方向/分镜」。两者合起来才是完整闭环。落点同上轮 `src/workbench/generationCanvas/agent/runStoryboardPlanner.ts` + `storyboardPlan.ts`（带 retry budget 状态机）。诚实边界见下，本轮只标方向，真落地需配合 judge 校准 + 用户拍 retry 烧额度上限。

## 🟡 架构教训（影响 Nomi 的设计决策，非即插即用）

### HollyWood Town / OmniAgent：把 decompose-stitch 的瀑布管线从「有向无环图」改成「有限重试的有向有环图」，让下游反馈能回改前面阶段（2025-10）
[arxiv 2510.22431](https://arxiv.org/abs/2510.22431) · [项目页 olpleo.github.io](https://olpleo.github.io/) · **未找到开源代码 repo**（项目页在、repo 搜不到）· HKUST + Stanford + University of Chicago（作者含 Anyi Rao / Maneesh Agrawala / Huamin Qu —— DirectorBench 同组团队）· 摘要未明示 training-free，但属黑盒多 agent 编排框架（cs.MA），机制不碰模型内部 · 日期：2025-10-28（窗内）。
- **它打的正是 Nomi 的架构**：摘要明说现有长视频系统要么「线性任务序列」、要么「中心化 director agent 管所有」，**两者都建在单向 DAG 上**，无法 ① 跨阶段共享正确上下文、② 一旦下游模块暴露不一致就回头反思修正。这就是 Nomi「拆镜头规划师→每镜生成」的单向链路。
- **三个机制（可分开借鉴）**：① **影视流水线式的分层 agent 图**（concept→剧本→分镜→资产→剧本监督→后期）；② **transient hypergraph（临时群聊节点）**——某 agent 上下文不够时临时拉相关 peer 开个「碰头会」补上下文，降单 agent 记忆负担；③ **核心可借鉴点 = DAG → 有限重试的有向有环图**：允许 bounded cyclic execution，下游节点的反馈能**回灌改前面阶段**（"improving earlier stages through feedback from subsequent nodes"），在 retry budget 内反复反思精化。
- **影响 Nomi 哪个决策**：Nomi 现在拆镜头→生成是**纯单向**（记忆 `storyboard-image-first-convergence` 的 shot→shot 链是往下的，崩了只在本镜局部修，回不到改分镜）。OmniAgent 的「有限重试有环图」给了一个明确范式——**镜级失败应能回灌给拆镜头规划师、允许改前面的分镜/剧本再重展开，但用 retry budget 封顶防死循环+烧额度**。这和 Nomi 记忆 `reconcile-edge-drop-and-card-redesign`（对账卡「让 AI 修」）是同向但更上一层：对账卡现在多是局部修，OmniAgent 说该允许「回头改更前面那步」。
- **诚实边界**：① 无开源代码，只能抄架构思想不能抄实现；② 摘要级信息（PDF 正文未解码成功），具体 retry 预算/收敛证据需后续读全文核实（待核实）；③ 它是「全自动拍片」定位、含一堆 agent，Nomi 不必全抄，只取「有限重试回灌改前阶段」这一个机制。

### MUSE：把长视频叙事重构成「闭环约束执行问题」，用 plan-execute-verify-revise 循环把叙事意图编译成机器可执行约束（2026-02）
[arxiv 2602.03028](https://arxiv.org/abs/2602.03028) · **无 repo / 无项目页**（仅 arxiv）· 作者 Wenzhang Sun 等（lab 未在摘要标明）· 摘要未明示训练与否 · 配 MUSEBench（reference-free 评测）· 日期：2026-02。
- **诊断与 Nomi 同**：摘要点名「intent-execution gap」——叙事意图要在长程逐镜多模态生成中存活，现有「feed-forward 管线或纯 prompt refine」会随序列变长产生**语义漂移 + 身份不一致**。这正是 Nomi 拆镜头后逐镜生成越往后越飘的问题。
- **机制 = plan-execute-verify-revise 闭环**：把叙事意图转成对**身份 / 空间构图 / 时序连续**的「机器可执行控制」，生成后用多模态反馈校验、违反即修。和 HollyWood Town 异曲同工（闭环 vs 单向），但把约束讲得更结构化（identity/composition/continuity 三类显式约束）。
- **影响 Nomi 哪个决策**：印证「叙事意图要落成结构化、可被校验的约束，而不是只塞进 prompt 文本」——和 Nomi 记忆 `memory-system-redesign-2026-06-20` 的核心判断（**结构化设定卡 > 自由文本，身份锁在生成参数**）完全同向。MUSE 给的增量是「这些约束生成后要被显式 verify、违反要 revise」，对接 Nomi 的对账/校验闸应往「逐镜对身份/构图/时序三类约束做机器校验」收口。
- **诚实边界**：无代码、无项目页、lab 不明、摘要级信息，**当设计参照不当落地源**；与上轮 Co-Director 的「全局方向注入」+ 本轮 HollyWood Town 的「回灌改前阶段」共同构成「闭环」三件套，三篇互相印证但只 Co-Director 有可用开源代码。

### DramaDirector：plot→storyboard 解耦「静态视觉条件（构图/机位）」与「动态叙事条件（动作/台词/情绪）」，几何（深度+姿态骨架）引导首帧（2026-06，4 天前）
[arxiv 2606.24107](https://arxiv.org/abs/2606.24107) · **开源 `github.com/iLearn-Lab/DramaDirector`**（star 数未核实）· 华南理工 + HKUST(广州) + 哈工大深圳 · **要训练**（Qwen3-8B LoRA SFT + GRPO + 对比学习的文-视对齐奖励模型）· 黑盒后端 Seedream-v5-lite/Vidu-q3-turbo/Tongyi 嵌入 · 日期：2026-06-23（窗内最新）。
- **可借鉴的设计点（非代码）**：① **每镜把「静态视觉条件（机位/构图/场景）」与「动态叙事条件（动作/台词/情绪）」解耦**——静态条件检索一张深度-姿态参考锚定首帧，动态条件驱动 i2v。这和 Nomi 的「3D 站位参考锚首帧 + prompt 驱动生成」（记忆 `staging-reference-tool-shipped`）**架构高度同构**，可借鉴「把镜头描述显式拆成静态/动态两类字段」这个 schema 设计。② **几何引导 = 从真实短剧镜头库按深度+姿态骨架预索引检索构图**（"保留构图布局、丢弃外观噪声"），思路与 Nomi 站位参考一致（用骨架/几何锁站位、不锁外观）。
- **为什么不进 🟢**：核心价值（planner + 对齐奖励模型）**必须训练**，要 DramaBoard 语料（35 剧/2.8K 集/81K 镜）+ 深度姿态预处理 + 画廊构建 + 两阶段训练；几何画廊是命门（消融去掉深度检索近乎崩）。对走 vendor API、solo、不训模型的 Nomi 不是 drop-in。
- **影响 Nomi 哪个决策**：印证「镜头条件该拆成静态(几何/构图)×动态(动作/叙事)两类」的 schema——可指导 Nomi 拆镜头规划师产出的镜头字段结构（把机位/构图字段和动作/情绪字段分开，前者接 3D 站位参考、后者进 prompt），但**不抄它的训练管线**。
